### DF126 — profile image tags carry no principal, so two principals with a same-named profile collide on one image (RESOLVED 2026-07-17)

- **Resolved 2026-07-17** — added `config.ProfileImageTag(layout, profileName)` (`internal/config/profile.go`), which returns `InstancePrefix(layout.Principal) + profileName` and so goes through D126's empty-principal panic. Routed all three hardcoding sites through it: `ResolveProfileImage` (read side), `profiles.EnsureProfileImage`'s `tag := ...` (write side, `internal/orchestrator/profiles/profile_build.go`), and `ProfileAdmin.Delete`'s cleanup-hint `image := ...` (`profile.go`). Added `config.BaseImage = "yoloai-base"` as the one deliberately-unscoped constant and replaced the three matching literals with it. Extended the existing v4->v5 `PrincipalRename` migrator (`internal/orchestrator/migrate_principal.go`) so `restampPrincipal` also re-stamps `Environment.ImageRef` from `yoloai-<profile>` to `yoloai-<principal>-<profile>` via the same `ProfileImageTag`, leaving `yoloai-base` records untouched — the same unmigrated-sandbox pass D126 already runs, so no second migration was needed. Filed a `## Unreleased` BREAKING-CHANGES entry (`docs/BREAKING-CHANGES.md`, "Profile image tags are principal-scoped").
- **Discovered:** 2026-07-17 · **Workstream:** surveying what else should ride v0.9.0; the archived plan's [open question 4](../archive/plans/cli-is-a-principal.md) left it unanswered and nothing ever filed it
- **Severity:** MEDIUM, latent — it needs two principals sharing one daemon, which nothing in this repo does today (same reachability as DF115 before D126). Consequence if reached: principal A's sandbox silently runs principal B's image. Per D126 decision (4) that is a **naming** matter, not a security boundary — the embedder chooses its principals — so this is filed on the namespacing axis, not as the B2 code-exec issue.
- **Description:** D62 scoped instance names by principal and left image tags unscoped; D126 scoped the CLI but did not reach tags. Three sites built the tag by hardcoding the prefix, none of them via `config.InstancePrefix`:
  - `internal/config/profile.go:274` — `ResolveProfileImage` returned `"yoloai-" + name` (the read side), falling back to `"yoloai-base"`.
  - `internal/orchestrator/profiles/profile_build.go:74` — `tag := "yoloai-" + name` (the write side).
  - `profile.go:274` — `image := "yoloai-" + name` (the delete/cleanup-hint side).
  So two principals each holding a profile named `web` both built and resolved `yoloai-web` in a shared daemon; the second build won and the first principal's sandboxes silently ran it.
- **This is the DF98 class exactly one artifact over, and D126's poka-yoke did not cover it.** D126 made `InstancePrefix` incapable of emitting a bare `"yoloai-"` (it panics on the empty principal) precisely so the hardcode could not be written. These three sites never called it, so they hardcoded the string directly and the gate could not see them. That is worth stating plainly: the poka-yoke guards the *function*, not the *string*, which is the limit DF98 already recorded when a lexical ban on the prefix was rejected. `ProfileImageTag` closes exactly that gap — a fourth caller that hardcodes the prefix is no longer possible without also bypassing a named function everyone else uses.
- **It was not a pure cache rename — it touched persisted state.** `store.Environment.ImageRef` (`store/environment.go:39`) records the resolved tag per sandbox and is read on the launch/restart path (`launch.go:355,877`; `lifecycle/restart.go:197`), with a legacy backfill to `"yoloai-base"` (`environment.go:184-188`). The migrator extension above re-stamps existing records so restart never resolves a tag a rebuild has orphaned.
- **Scope note.** `yoloai-base` is yoloAI's own artifact, byte-identical for every principal, and sharing it is desirable (one copy, one build) — it stays unscoped as `config.BaseImage`. Only *principal-authored* profile images are scoped, per `research/principal-isolation.md` §B2: *"any principal-authored build artifact (custom profile/image) lives in the principal partition with a principal-scoped image tag."* B2's wider question (may principals author profiles at all?) stays the embedder's, per D126(4).
- **Pointer:** `internal/config/profile.go` (`BaseImage`, `ProfileImageTag`, `ResolveProfileImage`); `internal/orchestrator/profiles/profile_build.go`; `profile.go` (`ProfileAdmin.Delete`, `ProfileAdmin.infoBase`); `internal/orchestrator/create/prepare_profile.go`; `internal/orchestrator/migrate_principal.go` (`restampPrincipal`, `restampedImageRef`); `store/environment.go:39,184-188` (`ImageRef` persisted + backfilled); `internal/config/names.go` (`InstancePrefix`, the gate these sites now go through); `docs/BREAKING-CHANGES.md`; `research/principal-isolation.md` §B2; D62; D126(4); DF98.

### DF133 — the provenance hook could not tell "I authored it" from "I never saw it", and blocked its own author (RESOLVED 2026-07-17)

- **Resolved 2026-07-17** — `AUTHORED_TOOLS` now counts a `Write`'s **file_path** (never its body) as provenance, which is the strongest kind there is: composing a file means knowing what it says. Two tests pin the split: authoring clears a citation, and a Write whose *body* merely mentions the doc does not — the latter being the exact hole that WRITE_TOOLS' exclusion from READ_TOOLS exists to close, since otherwise the edit under test clears itself.
- **Discovered:** 2026-07-17 · **Workstream:** writing `agent-failures.md` and `research/uncanny-valley.md` — the hook fired on the commit that created them
- **Severity:** LOW, but the shape is the interesting part: it is a **false accusation**, the failure mode D122 explicitly named as the one that gets a hook disabled (*"an accusation is what gets a hook disabled"*). It had a live, ordinary trigger — create a research doc, then wire it into an index or a sibling doc in the same session — which is simply how a doc gets written. It had not fired before only because no research doc had been authored since the hook landed on 2026-07-15.
- **Description:** `unread_citations` exempted the file *currently* being edited (`cited -= {Path(target).name}`) but nothing authored earlier in the session. `Write` is excluded from `READ_TOOLS` deliberately and correctly — writing a path is not reading it, and without the exclusion the Edit under test would satisfy the check by containing the very citation that triggered it. But that exclusion is about a write's **content**; it silently also discarded the fact that a write's **path** is evidence of authorship. So: `Write(research/uncanny-valley.md)` → cite it from `agent-failures.md` → *"nothing in this session opened that file"*. The only ways through were to drop the citation or open a file you had just composed.
- **The demand was absurd and the action it forced was not.** Told to open the file it had written ten minutes earlier, the agent did — and the read genuinely verified that the claim being made about the doc ("collects moments the interaction stopped resembling a human one in either direction") matched its actual ABOUTME. A gate can be wrong about its reason and right about its remedy; worth remembering before "fixing" one by deleting it.
- **Pointer:** `scripts/check_citation_provenance.py` (`AUTHORED_TOOLS`, `read_tool_inputs`, and the `target` exemption it complements); `scripts/tests/test_citation_provenance.py` (`test_authoring_a_doc_this_session_counts_as_knowing_it`, `test_a_write_whose_body_merely_mentions_the_doc_is_not_authorship`); D122; [DF129](#df129--the-provenance-hook-counts-naming-a-path-as-reading-it-so-a-long-session-launders-its-own-citations) (its opposite: the same hook is too *permissive* about Bash mentions — this one was too strict about authorship, and both come from the same substring-over-tool-inputs design).

### DF125 — D126 made every pre-upgrade orphan permanently unreapable; `system prune` never learned the old prefix (RESOLVED 2026-07-17)

- **Resolved 2026-07-17** by giving each sweep back the CLI's pre-D126 identity — the fix
  [cli-is-a-principal.md](../archive/plans/cli-is-a-principal.md)'s Risks section asked for and the
  implementation skipped. It is deliberately **two different things**, because the backends are:
  - **Label backends (docker, podman, containerd, apple) — permanent, no expiry.** `IsOrphanCandidate`
    now also claims an instance whose `com.yoloai.principal` label is empty, gated on the sweep running
    as the CLI's principal. This is not a compatibility shim: the label is a fact the backend recorded,
    so reading it stays exactly correct while the value simply stops occurring. It is D62's own rule —
    *"enumeration filters by label, not by name-string splitting"* — applied to the identity the CLI
    used to have.
  - **Tart — a heuristic, and the only part with an eventual expiry.** Tart records nothing (DF124), and
    the legacy name is provably ambiguous, so `legacyCLIVMName` claims `yoloai-<S>` minus the namespaces
    that have ever actually been minted (`yoloai-base`, `cli-*`, the test principals' `tNNNNNNN-`). Its
    retirement is tracked as [DF127](findings-deferred.md) with a **trigger, not a release**: a user may
    upgrade 0.8.0 → 0.10.0 directly and still hold legacy VMs, so dropping it on a version schedule would
    abandon exactly the people it exists for.
- **The safety argument, which is why this is not a sandbox-shredder.** Claiming legacy instances is only
  safe because a legacy-named instance is *provably debris* by the time a sweep can see one: `system prune`
  is gated below the current schema, and the v4→v5 migration stamps v5 only after **every** sandbox is
  migrated (any refusal returns before the stamp). Two things now hold that up mechanically rather than by
  comment: `classifySandboxes` also names an unmigrated sandbox's legacy instance into `known` (so a live
  sandbox that merely failed to migrate is protected even if the invariant slips), and
  `TestGateExemptionsAreClosed` pins the gate's exemption set to exactly `help`, `version`, `system migrate`
  — because exempting `prune` would run it below v5, where every LIVE sandbox still carries the legacy name
  and matches nothing in `known`.
- **Verified by execution (2026-07-17, real Docker, dry-run).** The same A/B that exposed the strand now
  flips: `yoloai system prune --dry-run` reports **both** `yoloai-cli-newprobe` and the pre-D126
  `yoloai-strandedprobe`, where before it saw only the former. The gate guard was verified the same way —
  annotating `system prune` as exempt makes both new tests fail and name it.

- **Discovered:** 2026-07-17 · **Workstream:** surveying what else should ride the v0.9.0 release; found by asking what the D126 migration does *not* walk
- **Severity:** MEDIUM, and **time-boxed** — it is a regression this release introduces, and the remedy has a natural expiry (see Disposition). Not data loss: the stranded instance keeps running or sits stopped, holding disk and — on tart — a capped VM slot, which is the scarce resource. The user's recourse is a manual `docker rm` / `tart delete` / `ctr containers delete` **forever**, because no yoloAI command can name it again.
- **Disposition:** FILED, not fixed. **[cli-is-a-principal.md](../archive/plans/cli-is-a-principal.md)'s own Risks section prescribed the fix and the implementation did not deliver it**: *"Mitigate: the migration is the only thing that may rename, it is idempotent, and `system prune` learns the old prefix for one release."* Idempotency landed; the prune half did not. **It must ship in v0.9.0 or not at all** — "for one release" is only meaningful in the release that causes the break; shipped later, the debris is already unreachable and the compatibility window has no one left to serve.
- **Description:** D126's migrator (`internal/orchestrator/migrate_principal.go`) enumerates work from `sandboxesRoot` — it renames the instance of every sandbox that still has a **sandbox dir**. An *orphan* is by definition an instance whose sandbox dir is gone (crash debris, or [DF113](#df113--destroy-frees-the-sandbox-name-while-leaving-the-instance-behind-so-the-next-start-adopts-a-vm-it-never-provisioned)'s destroy-frees-the-name path). Such instances are therefore never migrated, keep their pre-D126 `yoloai-<name>` identity — and every post-D126 sweep now filters them out:
  - **label-equality backends** (docker, podman, containerd's container sweep, apple since `da4d1106`): `instanceLabels` always stamped `LabelPrincipal`, so a pre-D126 CLI instance carries `com.yoloai.principal: ""`. `runtime.IsOrphanCandidate` (`runtime/orphan.go:26-30`) evaluates `"" == "cli"` → **false**.
  - **prefix backends** (tart `prune.go:31,36`): `HasPrefix("yoloai-mybox", "yoloai-cli-")` → **false**.
  - **containerd's netns sweep** (`prune.go:109,133`): requires the stripped name to carry `yoloai-cli-`; an old `yoloai-yoloai-mybox` netns does not.
  - **seatbelt is unaffected** — its identity is a path under its own `SandboxesDir()` and a *bare* dir name, which never carried the prefix (the same property that kept it out of DF115).
- **Verified by execution (2026-07-17, Linux + real Docker, dry-run).** Two identical orphans were planted, differing only in the principal label, and the real `yoloai system prune --dry-run` was run against them: `yoloai-strandedprobe` (`principal=""`, the pre-D126 shape) is **not** selected; `yoloai-cli-newprobe` (`principal="cli"`) **is** selected and reported as `container yoloai-cli-newprobe`. So the sweep is healthy and the omission is exactly the cross-version one. Docker was verified live; the tart/containerd/apple rows above are read from code, not executed.
- **The legacy identity is NOT recoverable from the instance name — proved, not assumed (2026-07-17).** The pre-D126 elided namespace `yoloai-<S>` **overlaps every principal namespace**, because a `SandboxName` may contain `-` while a `PrincipalSegment` may not. Verified by execution against the real grammars: `InstanceName("", "acme-probe")`, `InstanceName("", "cli-foo")` and `InstanceName("", "my-app")` are string-identical to principal `acme`+`probe`, `cli`+`foo`, and `my`+`app` respectively — every component a legal input the parsers accept. **D62's non-collision proof does not cover this**: *"the default value is forbidden as a real `PrincipalSegment`, so a real principal can never collide with the elided form"* establishes only that nobody can *register* `""`; the collision needs no such registration, only a hyphen in a sandbox name — and D62's own motivating example name is `my-app`. This is the third gap found in that proof's reach (DF115 found it silent on prefix *containment*; this is *segment ambiguity*). Note new-vs-new **is** disjoint — `principalSegmentRe` forbids separators, so the first `-` always delimits the principal — so D126's structural guarantee holds going forward; it is only legacy-vs-new that overlaps.
- **Shape of the fix (not yet decided), which the above splits in two.**
  - **Label backends (docker, podman, containerd, apple) — permanent and exact, no expiry.** `instanceLabels` has always stamped both labels, so `LabelSandbox` present **and** `LabelPrincipal == ""` is a *recorded fact* meaning "created by a pre-D126 CLI". Honouring it is not a compatibility hack and needs no sunset — it is precisely D62's own rule (*"runtime enumeration filters by label, not by name-string splitting"*) applied to a value that will simply stop occurring. Admit it **only** when the layout's principal is the CLI's.
  - **Tart — the one real gap, and it must NOT be fixed by admitting `yoloai-<name>`.** That prefix matches `yoloai-acme-*` too, i.e. it rebuilds DF115 by hand. Tart stores no labels (DF124, resolved by retraction: for a label-less backend *"their identity is the instance name plus the DataDir"*), so the name is the only signal — and **that resolution's reasoning holds only for names minted after D126.** It rests on *"principal namespaces are disjoint at the parse boundary, so the name already carries the owner"*, which is exactly the property legacy names lack: they predate the parse boundary and overlap it. DF124 is not wrong; its guarantee simply starts at D126, and this finding is the debris on the far side of that line. The pragmatic matcher — sufficient for *what has actually existed*, which is the only population there has ever been — is: legacy ⟺ `yoloai-<S>` excluding `provisionedImageName` (`yoloai-base`), `cli-*`, and the test-principal shape `^t[0-9]{7}-`. Integrator principals on tart have never existed; if one ever does, this over-reaches, so **this half is dated and deleted next release** while the label half is not.
- **Pointer:** `internal/orchestrator/migrate_principal.go` (`unmigratedSandboxNames` — what the migration walks, and does not); `runtime/orphan.go:26-30` (`IsOrphanCandidate`); `internal/orchestrator/launch/launch.go` (`instanceLabels`, which always stamps the principal, so pre-D126 = `""`); `runtime/tart/prune.go:31,36`; `runtime/containerd/prune.go:109,133`; `store/paths.go` (`LegacyCLIInstanceName`); [cli-is-a-principal.md](../archive/plans/cli-is-a-principal.md) § Risks; D126; [DF113](#df113--destroy-frees-the-sandbox-name-while-leaving-the-instance-behind-so-the-next-start-adopts-a-vm-it-never-provisioned) (which manufactures the debris this strands).

### DF115 — tart's and apple's prune match by name PREFIX, so the empty-principal CLI's sweep reaps every other principal's instances (RESOLVED 2026-07-17)

- **Resolved** across D126 and the apple label-equality fix. Fix (a) landed with D126: the empty principal is gone, so no namespace contains another and the live hazard the title names cannot be constructed. Fix (b) landed for containerd's `reconcileBlockingContainers` and now for **apple**, which lists `container list --all --format json` and filters through `runtime.IsOrphanCandidate` — verified against the live CLI 1.0.0 on Apple Silicon: labels ride at `configuration.labels`, so one list call suffices and the feared per-name `container inspect` fanout was never needed. **seatbelt** was struck as never affected (2026-07-16). **tart** is closed by decision rather than by code — see below. containerd's netns sweep keeps name-matching by necessity: a netns carries no label to compare.
- **The remaining rationale was wrong, and stating that is the substance of this closure.** The 2026-07-16 disposition kept the row open as "a belt against reintroduction", reasoning that prefix-matching is correct "only because no principal is a prefix of another, an invariant nothing enforces at the sweep". That invariant *is* enforced — at the parse boundary, which is where invariants belong. `principalSegmentRe` is `^[A-Za-z0-9]+$` (`internal/config/names.go:42`) and `InstancePrefix` appends a delimiting `-`, so `yoloai-cli-` cannot prefix `yoloai-clix-foo`: the character after `cli` is `x`, never the `-` the prefix requires. Distinct namespaces are structurally non-prefixing for every principal `ParsePrincipalSegment` can mint.
- **So the prescribed regression test proves nothing — measured, not reasoned.** The fix instruction was "plant an instance under a foreign principal and assert the sweep spares it". Built exactly that way against live hardware, that test **passes against the unfixed name-prefix predicate**, because two valid principals are disjoint by construction and both predicates therefore spare the foreign instance. Confirmed by mutating the predicate back to `strings.HasPrefix` and re-running: still green. A test that cannot fail is worse than no test — it certifies the belt while exercising the one thing that was never at risk. This is DF115's own lesson recurring: the finding twice mis-inferred behavior from a name (`InstancePrefix` in seatbelt's prune), and here it mis-inferred a hazard from a prefix.
- **What the fix actually buys** is the other half of `IsOrphanCandidate`, which the finding never emphasised: a container merely NAMED `yoloai-<principal>-*` that yoloai never created carries no `com.yoloai.sandbox` label, and the old predicate destroyed it — a hand-run `container run --name yoloai-cli-x` was collateral damage of `yoloai system prune`. That case is **live, not latent**, and it is the assertion `TestApple_PruneSparesForeignPrincipal` actually fails on when the predicate regresses. The cross-principal assertion is kept beside it as a belt, with a note saying it does not discriminate.
- **tart — closed by decision (2026-07-17, owner's call): for a label-less backend the name IS the identity.** Not implementable as specified, and a side index is not worth building. Tart has no native labels; the map reaches disk only as an incidental whole-struct `json.Marshal(cfg)` into `sandboxDir/backend/instance.json` (`tart.go:272,280`), and a tart orphan is *by definition* a VM whose sandbox dir is gone — the label store vanishes exactly when it would be needed, so D62's "store the map in the sandbox dir" is circular. The alternative, a yoloai-owned index keyed outside the sandbox dir, was weighed and rejected: it adds a persistence surface that must stay consistent with reality — a new class of drift — to guard a hazard that structurally cannot occur (see the paragraph above). Tart instance names are `yoloai-<principal>-<name>` with principals disjoint at the parse boundary, which is a sound identity. See [DF124](#df124--instanceconfiglabels-promises-owner-enumeration-that-tart-and-seatbelt-cannot-deliver-resolved-2026-07-17).

- **Docker is immune, and the reason indicts the others.** It filters by label **equality** — `runtime.IsOrphanCandidate`, `labels[LabelPrincipal] == string(principal)` (`runtime/orphan.go:26-30`) — so `""` matches only `""`. The research's Q4 "non-collision proof obligation" reasons entirely about name equality (`principal-namespacing.md`), so the gap was never in scope for it either. This is the same shape as DF98 — the empty sentinel makes wrong code work — except no one hardcoded anything here; the prefix is derived correctly and is still a superset.
- **Verified by execution on Apple Silicon, 2026-07-16** (dry-run only — a real unprincipaled prune here would have deleted the developer's `yoloai-embrace`, which is the defect demonstrating itself). ### DF122 — ~~reset baselined before checking for a SandboxSide backend, so a tart reset re-copied on the host and left the VM untouched~~ — NOT A BUG, disproven by the hardware test its own entry asked for (2026-07-16)

- **Discovered:** 2026-07-16 · **Workstream:** none — surfaced while consolidating the baseline call sites for DF120. **Retracted 2026-07-16** after the hardware test this entry named was actually run.
- **Severity:** none — there is no defect. Originally filed MEDIUM.
- **Disposition:** RETRACTED. The code change made under this number stays as a consistency cleanup (see below), but it fixes nothing user-visible, so nothing here is a fix.
- **The claim, and why it looked right on paper.** `resetCopyWorkdir` checked `IsGitRepo → HeadSHA` before the SandboxSide deferral, where create checks the deferral first. For tart the staging copy carries a `.git`, so `HeadSHA` returned a SHA, the deferral was skipped, and a non-empty SHA suppresses the `reset.go` gated call to `ExecuteVMWorkDirSetup` (`Mode == "copy" && BaselineSHA == ""`). That reasoning is correct as far as it goes — and it goes one call site too short.
- **Why it is wrong.** A tart reset does not just re-copy; it **recreates the container** (tart is SandboxSide, so `Reset` auto-upgrades to restart, which destroys the VM and boots a new one). And `recreateContainer` re-runs `ExecuteVMWorkDirSetup` **unconditionally** for copy mode — `restart.go`, comment: *"Always re-run when recreating: the old VM was destroyed, so its local work directory no longer exists even if BaselineSHA is already set."* So there are **two** VM-setup calls in a reset, and the unconditional one at `restart.go` always fires. The gated one at `reset.go` that this finding was about is redundant belt-and-braces for tart. The VM is refreshed on every tart reset regardless of what `resetCopyWorkdir` returns.
- **Disproven by execution on real tart, 2026-07-16.** `TestIntegrationTart_ResetRefreshesVMWorkDir` plants a marker inside the VM, moves the host source on with an upstream commit, resets, and checks the VM. It **passes against the pre-fix ordering** (85s): marker **GONE**, upstream **PRESENT**. The reset log shows `reset complete → starting sandbox → recreating container → launching container` — the recreate, and with it the unconditional refresh. If DF122 were real, the marker would have survived; it did not. This is the test the entry itself said would "settle the user-visible half", and it settled it against the finding.
- **What the unit test actually proved.** `TestResetCopyWorkdir_SandboxSide_DefersBaseline` asserts the returned SHA is empty and fails against the old ordering — true, but it tests the *mechanism* (does `resetCopyWorkdir` defer?), not the *consequence* (does the VM get stale?). The consequence never followed from the mechanism, because the second call site covers it. A green unit test on a mechanism is not evidence of the bug the mechanism was blamed for — the lesson DF98 keeps teaching.
- **The code change stays, as cleanup not fix.** `resetCopyWorkdir` now checks the deferral first and routes through `baseline.WorkCopy`, matching create and avoiding a throwaway SHA that `restart.go` overwrites anyway. ### DF117 — the default `yoloai reset` rsyncs the host tree verbatim, re-importing the `.gitignore`d secrets `:copy` exists to exclude and the history `--copy-strict` exists to strip (RESOLVED 2026-07-16)

  `CopyProjectDir` — the same dispatch create uses — and then prunes the copy to the file set
  `ProjectFileSet` names, so a reset reproduces the copy create would have made instead of
  approximating it with a second set of rules. Because reset runs create's own copy code, "reset
  honors the copy's semantics" stopped being a property to gate and became true by construction.
- **The design in the plan was not what got built, and the plan records why.** It kept rsync and
  fed it an allowlist (`--files-from`). Dropping rsync outright turned out to be strictly better:
  the whole class of pattern bugs disappears with the patterns, and so does a booby trap the
  investigation turned up — `rsync -a --files-from` does **not** recurse into a listed directory,
  but `rsync -a -r --files-from` **does**, so adding an `-r` that looks redundant next to `-a`
  would silently re-import the artifact trees. The premise for keeping rsync ("a differential
  sync never leaves a window where the tree has vanished") survives without it: `CopyDir` and
  `copyFileList` overwrite **in place** and never replace the destination directory, so the inode
  the container's bind-mount resolves to survives, which is the property that actually mattered.
  Verified with `os.SameFile`, and asserted by `TestResyncWorkCopy_PreservesWorkDirInode`.
- **Cost, accepted:** unchanged files are re-copied rather than skipped, so a reset now costs what
  a create costs. A size+mtime skip in `copyFile` would restore differential speed, but it is the
  same quick check that caused DF118, so it was not added on speculation.
- **Verified:** the six new `TestResyncWorkCopy_*` tests fail against the old rsync behaviour and
  pass after; the three that pass under both are exactly the things rsync already did right

  host runtime dependency: `pull-requests.md`'s required-tooling table and the two obsolete

### DF118 — in-place reset's rsync merges the host's `.git` into the work copy's instead of replacing it, and can leave refs pointing at deleted objects (RESOLVED 2026-07-16)

- **Resolved by deleting the merge, not by out-arguing it.** `resyncWorkCopy` removes the work
  copy's `.git` outright and lets `CopyProjectDir` put the source's back, so the two repos are
  never unioned and there is no ref for a quick check to skip. Removing it first also restores the
  CoW clone, which needs a fresh destination — so the correct thing is also the faster one here.
  The narrow same-second mtime coincidence this entry described is now unreachable rather than
  unlikely, which is the better kind of fixed.
- **Verified:** `TestResyncWorkCopy_ReplacesGitRatherThanMergingIt` resyncs a work copy whose
  baseline repo is unrelated to the source's — precisely the case that produced
  `fatal: bad object HEAD` — and asserts the resulting repo is readable and adopts the source's
  history. It fails against the old rsync behaviour.
- Fixed alongside DF117 because they were the same call site, and the `.git` half could not be
  made correct without answering this.

- **Discovered:** 2026-07-16 · **Workstream:** none — surfaced by a test written for DF116, which failed for this reason rather than its own.
- **Severity:** MEDIUM — real but narrow. Corrupting the work copy costs the agent's sandbox, not the user's repo, and the trigger needs a same-second mtime coincidence (below). Filed because the failure mode is silent and the recorded baseline outlives it.
- **Disposition:** PARKED — filed, not fixed. Shares a call site with DF117 and probably a fix; neither belongs in a bugfix branch.
- **Description:** `rsync -a --delete` reconciles trees **file by file**; it does not replace a directory wholesale. When the work copy's `.git` and the host's `.git` are different repositories, the sync unions them: `--delete` removes objects the host lacks, while each ref file is copied only if rsync's quick check says it differs. Ref files are **always 41 bytes**, so size never discriminates them and the decision rests entirely on mtime, at 1-second granularity. If a ref's mtime matches, rsync skips it — and the work copy keeps a ref pointing at an object `--delete` just removed.
- **Verified by execution, 2026-07-16** (pure rsync and git, no yoloAI code — so this is rsync's semantics, not a yoloAI bug in the usual sense). Built two unrelated repos, `orig` (HEAD `ef79a94`) and `work` (HEAD `b7bbd48`), each `git init` + one commit within the same second, then `rsync -a --delete orig/ work/`:
  - `work/.git` → still a directory, **merged** rather than replaced.
  - `git rev-parse HEAD` in `work` → `b7bbd48`, i.e. **work's own old ref survived** the sync.
  - `git log -1` in `work` → **`fatal: bad object HEAD`**. The ref points at an object that no longer exists.
  - `HeadSHA` returns that SHA **without error**, so `resyncWorkCopy` would record a baseline naming a missing object and `diff` would fail later, far from the cause.
- **Production reachability, honestly.** The coincidence is rare, and the common paths escape it: a `:copy` work copy's `.git` is a CoW clone of the host's, so their refs start identical, and any divergence (agent commits, or the user commits) moves one ref's mtime and rsync then copies. The exposed case is a work copy whose `.git` is structurally unrelated to the host's — `--copy-strict`, or a fresh baseline — resynced within a second of the host's last ref write. The reproduction hit it instantly only because a test does everything in the same second. **Not verified against a real backend**; the check that would settle how often this bites is a `--copy-strict` sandbox reset in a loop against a repo committed to concurrently.

### DF111 — in-VM git fails exit 69 because tart setup switches `xcode-select` onto the mounted Xcode *before* accepting its licence — ADDRESSED-IN-PLACE (root cause found; the original diagnosis was wrong)

- **Discovered:** 2026-07-15 · **Workstream:** releasetest on Apple Silicon (post-DF94 rerun); root-caused and fixed on the Mac 2026-07-15
- **Severity:** MEDIUM — a **shipped** bug, not a test bug. D113 routes copy-mode work-copy git **inside** the VM for tart, so real agent work sits in the same window the tests caught. Measured incidence **1 failure in 10** runs of `TestIntegrationTart_GitCorruption`. Not CRITICAL: the window is milliseconds wide in practice, it fails closed (git errors, nothing is corrupted), and production's baseline path already retries around it.
- **Disposition:** ADDRESSED-IN-PLACE — accept the licence **before** the switch, scoped with `DEVELOPER_DIR`. Verified against a live reproduction: **1/10 failures before, 0/40 after** (p ≈ 0.015 that 40 clean runs happen by chance at the observed 10% rate), with zero exit-69 occurrences.
- **Root cause.** `TartBackend.setup` ran `sudo xcode-select --switch <mounted host Xcode>` and only *then* `sudo xcodebuild -license accept`. Apple's `/usr/bin/git` is a shim that resolves the **active** developer dir and exits 69 ("You have not agreed to the Xcode license agreements") whenever that dir is an Xcode this VM has not accepted. Between those two calls every in-VM git fails. The window cannot be closed by provisioning: the acceptance record lives in the VM's own `/Library/Preferences`, so it dies with each ephemeral VM (DF112), and the base image cannot pre-accept because the record is **version-keyed** and the host's future Xcode version is unknowable at build time. Verified on this host: Xcode **26.5**, `IDEXcodeVersionForAgreedToGMLicense` = **26.4** (the host itself had not accepted 26.5), and every fresh VM carried **no** record at all — so the accept is load-bearing on every single start.
- **Why callers land in it.** The host does not wait for this script when no secrets are staged: the barrier at `launch.go:802` is `if hasSecrets`, and `hasSecrets` is literally `secretsDir != ""` (`launch.go:128`) — false for the tier's agent `test`. That barrier's own comment says it exists to stop the host deleting the secrets temp dir before the guest reads it; it is **not** a "setup finished" barrier, and only incidentally lands after `setup()` when it runs at all.
- **The fix.** `sudo env DEVELOPER_DIR=<mounted Xcode> xcodebuild -license accept` **first**, then the switch. `DEVELOPER_DIR` out-ranks `xcode-select` for the shim, so the licence is accepted for the target Xcode while the active dir is still the CommandLineTools git that already works — after which the switch cannot open a gap. Verified in isolation before the loop: on a fresh clone with Xcode mounted `:ro` as the backend mounts it, `switch` → `git` = **exit 69** and `git add` = **exit 69**; with accept-first, git = **exit 0** across the switch and the record reads 26.5.
- **The original diagnosis was wrong, and both its open questions are answered.** It read "the tart base VM's git needs an Xcode licence agreement" and asked whether the acceptance was missing from `yoloai-base` provisioning or inherited from host state. **Neither.** A raw `tart clone yoloai-base` + `tart run` runs `git --version`, `git init`, `git status` and `git add` all at **exit 0**, with `xcode-select -p` = `/Library/Developer/CommandLineTools`, which needs no licence. The base VM's git is fine; the breakage is introduced by yoloai's own setup, seconds later. And `git --version` **does** trigger it — but only once the active dir is an unlicensed Xcode, which is why probing a raw base VM shows nothing and sends the reader toward the base image.
- **Three wrong turns, recorded because each was defensible and each cost real time.**
  1. **The storm.** `backend-idiosyncrasies.md` attributes this exact string (it is in its Symptom Index) to the `xcodebuild -runFirstLaunch` security-scan storm. AGENTS.md says read that file *before* diagnosing a backend problem; skipping it meant re-deriving an answer the file already had a competing entry for. But the entry does not hold here: backgrounding a real `-runFirstLaunch` while probing once per second gave **90/90 exit 0** for git and zero misses for both `which tmux` and a direct stat of `/opt/homebrew/bin/tmux` — its *primary* symptom. Firstlaunch measured **5s and 8s** against its "60-120s+", and its persistence premise is false (DF112).
  2. **"The window is unreachable."** Widening the window to 25s with a temporary sleep and running the test **passed**, which read as proof no caller enters it. It was the opposite of proof: the sleep delays the **accept**, not the switch, so it pushes callers *past* the accept and converts a 10% flake into a certain pass. A negative result from an instrument that suppresses the signal is worth nothing, and this one nearly retired the correct diagnosis.
  3. **"It is not reproducible."** The tier passed ~5 consecutive times, then 9 in a row, and both were read as evidence. At a 10% rate that is unremarkable sampling. What settled it was looping the single test 20× — the failure landed on run 10, verbatim. **A green tier proves nothing about a 10% flake**; only a loop does.
- **What ruled out the rival mechanism, rather than argued against it.** DF114's returncode check was landed in the same commit, so a failed accept now logs `tart.xcode.license.failed` with stderr instead of claiming success. ### DF68 — `system migrate` is not crash-safe (stamp-before-pass; no journal/atomic-commit/lock/rollback) — RESOLVED

- **Discovered:** 2026-06-30 · **Workstream:** overlay-retirement migration design (the multi-step overlay→copy flatten forced the question)
- **Severity:** MEDIUM-HIGH (a crash/kill during `system migrate` can leave the data dir inconsistent — a green stamp over half-migrated data, or a partially-transformed sandbox — with no automated recovery and possible data loss; a *persistent* migration bug can brick the user with no escape)
- **Disposition:** **RESOLVED 2026-07-01** — built on branch `crash-safe-migration` (D110): `internal/fileutil` durable primitives + `internal/migrate` crash-safe promotion with an exhaustive 8-name-set recovery classifier (stamp-last, truth = on-disk form) + a `Migrator` plan/apply gate. - **Description:** `System.MigrateDataDir` calls `config.MigrateLibrary` — which `WriteSchemaVersion`-stamps the realm to current — **before** running the per-sandbox pass `orchestrator.MigrateAgentConfigs`. ### DF70 — host `git apply --unsafe-paths` of a container-generated patch = arbitrary host-file write (path traversal) — RESOLVED

- **Discovered:** 2026-06-30 · **Workstream:** crash-safe-migration re-audit (security reviewer; the overlay→copy flatten made the apply path load-bearing) · **Verified in code 2026-06-30** (maintainer) + reviewer reproduced the traversal with git 2.43.0
- **Severity:** CRITICAL (arbitrary host-filesystem write as the host user → RCE via `~/.ssh/authorized_keys`, `~/.gitconfig` `core.fsmonitor`/alias, or `~/.yoloai/profiles/base/Dockerfile` on next build). Was reachable in shipped code, not only in the unbuilt migration.
- **Disposition:** **RESOLVED 2026-07-01, commit 35c8c576** (landed on `main`, independent of the overlay-retirement work; rebased into `release-prep`). The apply-side sibling of DF66/DF67 (C1), which confined only the diff-**generation** side; the host-side **apply** of the resulting untrusted patch had been declared "sound because `--unsafe-paths` is gated" — that conclusion was **wrong** (`--unsafe-paths` is not path-confining). The non-repo apply now runs `git apply` with **cwd=target and `--git-dir` pointed at a throwaway temp repo**, dropping `--unsafe-paths`/`--directory=` entirely (new `applyNonRepo`; `runGitApply` split so a `--git-dir` global flag can precede the subcommand via new `runGitInput`). The explicit git-dir blocks ambient-repo discovery so the target itself is the working area, and without `--unsafe-paths` git rejects any escaping path (`../`, absolute, `.git/`, beyond-symlink). One change covers both `ApplyPatch` and `CheckPatch`, and thus both the overlay apply and copy-mode's non-git `ApplyAll` branch (both reach it via `IsGitRepo(hostPath) == false`). Empirically verified on git 2.43.0: traversal patches rejected (exit 128, nothing written outside the target), legit in-target patches still apply — including when the target sits under an unrelated ambient repo. `make check` green on both branches.
- **Description:** The overlay patch is produced by `git diff` run **inside the untrusted, agent-controlled container** (`copyflow/apply.go:495-520`, `generateOverlayPatchForContext` → `execInSandbox`), so its bytes are attacker-controlled (the agent has passwordless sudo and can shadow `git`). That patch was applied **host-side** by `git.NewHost(...).ApplyPatch(...)`; for a **non-git** target (`:overlay`/`:copy` dirs aren't repos) it routed through `git apply --unsafe-paths --directory=<target>` from a throwaway temp repo created **elsewhere** (`withTempGitDir`) with writes redirected via `--directory=`. Because the target was outside that temp repo's worktree, git's "refuse to write outside the working tree" guard would fire, so `--unsafe-paths` was added to switch it off — and a `../`+absolute patch path then wrote anywhere the host user could. The old in-code comment claimed the premise that made it "safe" (patches "produced by our own git diff over the target tree … never an agent-supplied raw patch") — **false for the overlay path.**
- **Pointer:** `internal/git/ops.go` (`applyNonRepo`, `runGitInput`, `CheckPatch`, `ApplyPatch`), tests `TestApplyPatch_NonGitDir_RejectsTraversal` / `TestCheckPatch_NonGitDir_RejectsTraversal` / `TestApplyPatch_NonGitDir_AmbientRepoDoesNotWiden` in `internal/git/ops_test.go`; commit 35c8c576. Refutes the soundness claim in [plans/copy-mode-git-rce.md](../archive/plans/copy-mode-git-rce.md) (C1/DF66).

### DF37 — File-locking hardening: confirm `flock` not `fcntl`, add the fsync-durability dance

- **Discovered:** 2026-06-15 · **Workstream:** public-layering (persistence helper, D87)
- **Severity:** MEDIUM (silent-corruption / data-loss avoidance)
- **Disposition:** PARKED (verify + harden with the persistence helper)
- **Description:** Two file-locking footguns the research flagged: (1) **`fcntl`/`F_SETLK` POSIX record
  locks release when *any* fd to the file is closed** (a library that opens/reads/closes the same file
  silently drops the lock — the man page calls it "bad") and don't inherit across `fork` as expected;
  **`flock`** has sane semantics (binds to the open file description, self-cleans on crash) and is
  portable to macOS. Confirm `store/lock_unix.go` (and any other locking) uses `flock`, not `fcntl`.
  (2) Atomic `rename` gives atomicity but **not durability** — a crash can leave a zero-length file
  (the real ext4 delayed-allocation bug) unless the write does **`fsync(temp) → rename → fsync(parent
  dir)`**. Add that dance to the atomic-write path.
- **Pointer:** `store/lock_unix.go`; `internal/fileutil` (atomic write). Design: [research/shared-state-concurrency.md](research/shared-state-concurrency.md) / [D87](../decisions/working-notes.md).

### DF36 — Persistence is unsafe on a network filesystem; detect and warn/refuse

- **Discovered:** 2026-06-15 · **Workstream:** public-layering (persistence helper, D87)
- **Severity:** MEDIUM (data-safety)
- **Disposition:** PARKED (implement with the persistence helper)
- **Description:** The persistence model (D87) relies on POSIX advisory locking (`flock`) + atomic
  `rename`. Both are **unreliable on network filesystems** — "POSIX advisory locking is known to be
  buggy or even unimplemented on many NFS implementations," and a lock that gets lost can lead to
  silent corruption ([sqlite.org/lockingv3](https://sqlite.org/lockingv3.html); man7 `fcntl_locking`).

  memory), so this is not a JSON-vs-SQLite escape — it's an environmental constraint. yoloAI should
  **detect the data dir's filesystem and warn (or refuse) on a network FS** rather than corrupt
  silently.
- **Pointer:** wherever the data dir is resolved (`config.Layout`); the persistence helper's open path. Design: [persistence-helper.md](persistence-helper.md) / [research/shared-state-concurrency.md](research/shared-state-concurrency.md) / [D87](../decisions/working-notes.md).
- **Update (2026-06-29):** RESOLVED — the lock path's filesystem is probed at acquisition (store/netfs*.go) and a one-time slog.Warn fires on a network FS (NFS/SMB/9P/AFS/FUSE; warn-not-refuse). Commit 46c52aa9.

- **Discovered:** 2026-06-09 · **Workstream:** nolint-exception audit
- **Severity:** MEDIUM
- **Disposition:** RESOLVED 2026-06-09. - **Description:** In `container-enhanced` (gVisor) isolation, `state.Perms` returned `SecretsDir: 0755` / `SecretsFile: 0644`, and `provision.CreateSecretsDir` stages the secrets via `os.MkdirTemp(stagingRoot, "yoloai-secrets-*")` where `stagingRoot` defaults to `""` → `os.TempDir()` (`/tmp` on Linux). `/tmp` is world-traversable, so for the window between staging and removal (the entrypoint reads `/run/secrets`, signals the consumed marker, then the host `RemoveAll`s the dir — up to `SecretsConsumedTimeout`, 30s default), **any local user on a multi-tenant Linux host could read the API keys / OAuth tokens** (`ANTHROPIC_API_KEY`, `CLAUDE_CODE_OAUTH_TOKEN`, `CODEX_API_KEY`, …). The original `perms.go` comment claimed the broad bits were a real gVisor constraint (gofer/pre-remap container UID must read host-created files); the empirical validation below disproved that.
- **Empirical validation (2026-06-09, real Linux + gVisor host):** Run on native Linux (KVM guest, Ubuntu 6.8.0), Docker daemon with `runsc` (release-20260309.0) registered. Mirrored yoloAI's Docker gVisor invocation: `docker run --runtime=runsc --user <hostUID>` (Docker has no `UsernsProvider`, so `UsernsMode=""`; `store.ContainerUser` returns the host UID as a numeric string for `container-enhanced` — the sandbox process runs as the *invoking host UID*, here 1000). Secrets staged on host owned by uid 1000, bind-mounted `:ro` to `/run/secrets`.

  | Variant | parent / dir / file perms | sandbox read (uid 1000)? | second host user (`nobody`) read? |
  |---|---|---|---|
  | **C** (tightest) | `0700` / `0700` / `0600` | **OK** | blocked |
  | **B** | `0700` / `0755` / `0644` | OK | blocked (by 0700 parent only) |
  | **A** (status-quo, `/tmp`) | `/tmp` / `0755` / `0644` | OK | blocked (by 0700 parent only) |

  **Gofer UID:** `runsc-gofer` and `runsc-sandbox` both run as **host root (uid 0)** — the host-side filesystem access is done by root, so the parent-dir mode never blocks the gofer.

  **Mechanism (negative + positive control):** gVisor nonetheless enforces *guest-side* uid/mode faithfully. With a `0600` file owned by host uid 1000: a sandbox running as **uid 1234 → `Permission denied`** (NEG); running as the **matching uid 1000 → reads OK** (POS). A `0750` host-owned dir is likewise writable by the matching-uid sandbox. So access is granted by *owner-uid match between the staged path and `ContainerUser`*, not by the root gofer bypassing perms, and not by world bits.

  **Caveat for future work:** the registered Docker runsc path uses `--user <hostUID>` + `UsernsMode=""`, which is what was validated. If a Podman-rootless gVisor path is ever added, re-validate, since `keep-id` maps the container to a subuid and the owner-match argument must be re-checked.
- **Pointer:** `internal/sandbox/state/perms.go`; `internal/sandbox/provision/provision.go` (`CreateSecretsDir`, `EnsureContainerSettings`); call sites `internal/sandbox/create/create.go`, `internal/sandbox/lifecycle/reset.go`, `internal/sandbox/launch/launch.go`.

### DF10 — `canCreateNetNS` leaked Go OS thread netns; libcni plugin execs landed in wrong namespace

- **Discovered:** 2026-05-26 · **Workstream:** containerd backend reliability (follow-up to DF9)
- **Severity:** HIGH (caused every observed "DF9" smoke failure; ~20% per-create failure rate in a tight loop)
- **Disposition:** ROOT-CAUSED + FIXED (2026-05-26).

- **Description:** `runtime/containerd/containerd.go::canCreateNetNS` (capability probe called on every containerd-backend `new`) called `netns.NewNamed(probe)` and `netns.DeleteNamed(probe)` with no `runtime.LockOSThread` and no `netns.Set(origNS)` restore. `NewNamed` calls `unshare(CLONE_NEWNET)`, which switches **the current OS thread** into a brand-new netns; after `DeleteNamed` removes the bind mount, that thread is in an anonymous netns. Without LockOSThread the goroutine can be scheduled off that thread, leaving it in Go's runtime pool **still in the wrong netns**. Any later goroutine landing on the poisoned thread inherits the netns — including libcni's `exec.Command` for plugin invocations. Bridge or firewall plugin then ran in the wrong netns and wrote iptables rules to a namespace the host can't see.

- **Symptom signatures (all observed):**
  - POSTROUTING entry for sandbox IP present in host, CNI-FORWARD ACCEPT for same IP missing (firewall on poisoned thread). Originally misattributed to upstream firewall plugin no-op (DF9 v1 evidence in smoke run `175645.907`).
  - POSTROUTING missing for sandbox IP, CNI-FORWARD present (bridge on poisoned thread). Captured in smoke run `194842.389/stop_start/containerd-vm/attempt1/network-diag.txt`.
  - `iptables -S CNI-FORWARD` returns `No chain/target/match by that name` even though `n.Setup` reported success (firewall created the chain in the leaked netns; that netns is anonymous so no other process can reach it).
  - `result.Interfaces["eth0"].IPConfigs` empty after `n.Setup` (libcni's result-build path returning a malformed result when an upstream plugin ran in the wrong netns).

- **Reproduction:** 20-iteration loop of `sudo -E ./yoloai new sb-$i /tmp/dir --agent test --os linux --isolation vm --yes --debug` in a session that has already run containerd `system check`. Pre-fix: 4/20 failures + 6/20 with wrong-netns observed in instrumented `iptables` subprocess. Post-fix: 20/20 success, 0 wrong-netns observed.

- **Fix:** Wrap `canCreateNetNS` in the same pattern `createNetNS` already uses — `goruntime.LockOSThread()` + `defer Unlock`, save `origNS` via `netns.Get()`, run the probe, `netns.Set(origNS)` to restore the thread's netns before unlock. Same single-callsite change.

- **Why DF9's mitigation masked this for a while:** The verify+retry path in `cni.go::setupCNI` sometimes landed on a clean thread on the retry, and the sandbox came up. The retry was attributed to "upstream firewall plugin bug" rather than "we have a thread netns leak". DF9's mitigation now stays as defense-in-depth — if it ever fires post-DF10, there is either (a) an actual upstream bug, (b) a different netns leak we haven't found yet, or (c) genuine iptables-nft transient state — and all three warrant investigation rather than another retry.

- **Pointer:** `runtime/containerd/containerd.go::canCreateNetNS`; entry "Go OS thread netns leak from `netns.NewNamed` / `netns.Set` without `runtime.LockOSThread`" in `backend-idiosyncrasies.md`. Cross-ref DF9 (mitigation kept as defense-in-depth) and DF8 (Kata warm-up race, separate cause).

### DF8 (11th data point, 2026-05-26): **SECOND SMOKING GUN — staged probe pinpoints the broken CNI stage**

- Log `yoloai-smoketest-20260526-150145.945`. Two `containerd-vm` failures (full_workflow + stop_start, both first-attempt then passed retry). Both show the identical staged-probe signature:
  ```
  network: unreachable [dns failed | dns=fail route=ok tcp=fail https=exit 28]
  ```
- **Translation:**
  - `route=ok` — CNI bridge plugin ran, IPAM assigned an IP, default route inserted into the netns.
  - `dns=fail` + `tcp=fail` — packets going OUT of the netns silently dropped. UDP query to the nameserver and TCP SYN to `1.1.1.1:443` both produce no response (timeout, not refused).
  - `https=exit 28` — confirms total outbound dead, same as the TCP probe.
- **Locating the broken stage:** the netns IS wired, the route IS pointing the right way, but packets aren't actually reaching the upstream. For Kata-VM specifically (which both failures here are), `backend-idiosyncrasies.md` documents the architecture: Kata creates a `tap0_kata` TUN/TAP inside the netns and installs a TC mirred filter that mirrors traffic between `eth0` and `tap0_kata`. The filter is what carries packets between the VM (via TAP) and the bridge (via veth/eth0). If the TC filter isn't fully installed when the agent's first packet fires, packets go in but don't come out — exactly what we see.
- **Confirmation that this is a race, not a deterministic break:** retries pass within 30s. The TC filter installation completes during the retry window.
- **Proposed fix location:** `runtime/containerd/cni.go::setupCNI` (or a post-`NewTask()` hook in `lifecycle.go::Create`). Two viable approaches:
  1. **Connectivity probe after CNI ADD + task.Start**: run a brief in-netns ping/TCP-connect to the gateway or upstream before declaring the sandbox ready. Fail-fast or short retry loop.
  2. **Post-Start sleep + verify**: short stabilization delay (similar to the existing "Tart exec needs brief stabilization delay after boot" pattern documented in backend-idiosyncrasies.md), then verify connectivity once.
- Approach (1) is more robust (catches deterministic CNI breakage too). Approach (2) is simpler but doesn't surface the real failure cleanly if connectivity NEVER comes up. Both should add a `backend-idiosyncrasies.md` entry describing the race.
- **DF8 family is now fully diagnosed.** Closing out further data-collection diagnostic work; the next step on this front is the fix.

### DF40 — seatbelt/tart launch left the host process attached to the caller's controlling terminal (macOS terminal-corruption sibling of the fixed docker bug) (RESOLVED 2026-06-29)

- **Discovered:** 2026-06-18 · **Resolved:** 2026-06-29 · **Workstream:** terminal-corruption investigation (the docker/Linux half was fixed earlier on `main` in `f208b32`)
- **Severity:** MEDIUM (terminal corruption / usability; **seatbelt and tart only**, i.e. macOS) · **Disposition:** RESOLVED (`34ccaac1`, verified on macOS)
- **Root cause.** Each macOS backend started its long-lived host process — `sandbox-exec … python3 sandbox-setup.py` (seatbelt) / `tart run` (tart) — with `SysProcAttr{Setpgid: true}`. `Setpgid` makes a new process group but stays in the **same session**, so the detached host process kept the caller's **controlling terminal** (`/dev/tty`) and could leave it in raw mode after a non-interactive `new`/`start` returned.
- **Fix.** Both launches now use `SysProcAttr{Setsid: true}` (new session, no controlling tty), mirroring the broker sidecar's detach idiom (`internal/broker/host.go`). Pointers: `runtime/seatbelt/seatbelt.go` Start, `runtime/tart/tart.go` Start.
- **Verification (macOS, Apple Silicon, macOS 26.5.1).**
  1. **Mechanism (PTY reproduction).** A detached child launched with `Setpgid` stays in the caller's session and retains `/dev/tty` (`ctty=HAS-CTTY`); with `Setsid` it is its own session leader with **no controlling terminal** (`ctty=none`) even with the PTY inherited on stdin — so it structurally cannot mutate the caller's tty.
  2. **Real non-interactive runs.** `yoloai new --backend seatbelt` and `--backend tart`, each driven under a PTY pre-set to cooked mode: both returned cleanly with the terminal **uncorrupted** (`raw_mode_left=False`); tmux came up live in each sandbox (`tmux ls` → `main` session via `yoloai exec`); and the host process was reparented (`PPID=1`) with **no controlling terminal** (`TTY=??`).
  3. Seatbelt + tart integration suites pass; `make check` green on macOS.
- **Teardown confirmed unaffected.** Seatbelt's `killByPID` does `syscall.Kill(-pid, …)` group-kill and the Setsid child is its own process-group leader, so the group is still reaped. Tart's teardown (`stopVM`) is independent of session/group entirely — it stops the VM via `tart stop` and discovers/`SIGTERM`s the `tart run` host process by PID via `pgrep` — so Setsid changes nothing there. (Note: the inline comment added to `tart.go` describing teardown as `kill(-pid)` group-kill is inaccurate for tart — that idiom is seatbelt's; tart's tmux/agent run inside the guest VM, not as host process-group children. Corrected in a follow-up.)

### DF88 — the `skip_upload` template resolved to `false`/`true` but never `auto`, so an `-rc` tag would publish the prerelease cask over the stable one

- **Discovered:** 2026-07-14 · **Workstream:** node20 action-runtime clearing (post-v0.8.0)
- **Severity:** MEDIUM · **Disposition:** RESOLVED (fix correct, and now verified live by the v0.9.0-rc.1 rehearsal — see below)
- **CORRECTION (2026-07-15): it was not latent. It had already fired, in production.** This line previously read "latent; no `-rc` tag has ever been cut, so it never fired — but it fires on the *first* one". Every clause of that is false. A `v0.7.0-rc1` tag *was* cut on 2026-07-06, it *did* publish the prerelease cask over the stable one, and `brew install kstenerud/tap/yoloai` served **0.7.0-rc1** to every user for about two hours fifty minutes — from `kstenerud/homebrew-tap@9c664e93` (08:21:22Z, cask body `version "0.7.0-rc1"`) until `@c440aa46` (11:11:33Z) replaced it with v0.7.0.
- **Why the original claim looked true.** The rc tag and its GitHub release were deleted after the fact, so the two places anyone would check — `git ls-remote --tags` and the releases list — show no rc and never did. The evidence survives only in the *tap's* git history, which is a different repository and the one place nobody looked. Deleting the tag erased the record from where it was searched for, not from where it mattered. This is D119's rule (the primary source for a claim about the past is the artifact, not the index of it) and D123's shape (a confident claim, refuted by a source one command away) arriving together — in an entry already filed as RESOLVED, where nothing would have revisited it.
- **Verified live 2026-07-15.** `v0.9.0-rc.1` (the DF89 rehearsal) exercised the fixed guard: the tap's newest commit is still v0.8.0 and the cask still serves `version "0.8.0"` — the rc did not touch it. That is the first time this guard has been observed working rather than reasoned about.
- **Root cause.** `release.prerelease: auto` correctly marks an `-rc`/`-beta` tag as a *GitHub* prerelease, but that setting governs only the release — it has no bearing on the tap. The cask's guard was `skip_upload: '{{ if .Env.HOMEBREW_TAP_TOKEN }}false{{ else }}true{{ end }}'`, which encodes exactly one question ("is the tap token configured?") and so resolves only to `false` or `true` — **never `auto`**. The cask publisher skips on `skip_upload == "true"`, or on `skip_upload == "auto"` **and** a non-empty semver prerelease; `false` matches neither. So with the token configured (it is), an rc tag would have published the rc cask into `kstenerud/homebrew-tap`, overwriting the stable cask — `brew install kstenerud/tap/yoloai` would then serve the rc to every stable user.
- **Fix.** `skip_upload: '{{ if .Env.HOMEBREW_TAP_TOKEN }}auto{{ else }}true{{ end }}'` — `auto` instead of `false` in the token-present branch. This keeps all three behaviours distinct rather than trading one for another: no token → `true` → always skip (a release can still be cut before the tap/secret exist); token + prerelease → `auto` + non-empty prerelease → skip; token + stable → `auto` + empty prerelease → publish. A bare `skip_upload: auto` would have been wrong — it would have dropped the token-absence guard.
- **Verified against upstream source, not the docs.** GoReleaser's docs describe `auto` but never state the default, and the schema declares `skip_upload` as `string|boolean` with no default — so the behaviour was read directly from `internal/pipe/cask/cask.go` (`doPublish`). `internal/pipe/brew/brew.go` has byte-identical logic, so a future `brews` formula needs the same template.
- **Pointer:** `.goreleaser.yaml` (`homebrew_casks[0].skip_upload`) vs `release.prerelease: auto`. Related: DF89 (the rc validation this unblocks).
