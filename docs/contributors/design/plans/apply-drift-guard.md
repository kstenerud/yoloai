> **ABOUTME:** Design draft, not a locked decision or implemented, for closing the gap where
> `yoloai apply` can silently land sandbox changes made after the user reviewed `diff` but
> before `apply` ran, by fingerprinting the reviewed artifact and balking on mismatch.

# Apply drift guard

**Status:** Design draft from a design conversation; not yet a locked decision, not
implemented. Key points below are marked confirmed-with-user vs. still-open.

## Problem

`yoloai diff` and `yoloai apply` are both computed **live**, independently, from
current git state — neither is cached (`copyflow/diff.go`, `copyflow/apply.go`).
The only thing persisted per tracked dir is `BaselineSHA`/`InceptionSHA`
(`store.DirEnvironment`), never diff content. This means a user who runs `diff`,
reads it, and then runs `apply` moments later has **no guarantee** that what
lands is what they reviewed: if the agent is still running, it can add commits
or edit files in the gap between the two commands, and `apply`'s default mode
(land everything beyond baseline) will pick up anything new right along with
what was reviewed — silently.

`diff` already prints an advisory warning when the sandbox is `active`/`idle`
("diff may be incomplete"), but it's cosmetic: it doesn't block anything, and
`apply` carries no equivalent check at all today.

## Core mechanism

Fingerprint the **exact artifact `apply` will land**, not the diff text `diff`
prints for humans. Store it per tracked dir when a plain `diff` is run; check it
right before `apply` lands changes; balk unconditionally on mismatch.

**Why not just hash the printed diff text:** `apply`'s default (commit-
preserving) mode lands a `git format-patch` series via `git am --3way`
(`copyflow/apply.go` `GenerateFormatPatch` → `internal/git/ops.go`
`ApplyFormatPatch`), while `--no-commit` mode lands a `git diff --binary`
blob via `GeneratePatch` → `ApplyPatch`. Neither is byte-identical to the
human-readable text `GenerateDiff` prints (binary hunks in particular render
as a placeholder in `diff` output but as real encoded bytes in `--binary`
patches). Hashing the pretty-printed diff and comparing it against a
freshly-generated apply artifact would produce false mismatches. So each
apply mode's fingerprint must be produced by **the same generator function
apply itself calls** — not derived from what the user was shown.

Two fingerprint kinds, one per apply mode:

1. **Commit-scope fingerprint** (protects the default, commit-preserving apply
   path): the ordered list of beyond-baseline commit SHAs
   (`git log --format=%H <BaselineSHA>..HEAD`). Commits are already
   content-addressed, so comparing the SHA list *is* comparing content — no
   hashing of patch bytes needed, and it's cheap (`git log`, no patch
   generation) to both produce at `diff` time and re-check at `apply` time.
2. **Worktree-scope fingerprint** (protects `--no-commit`/`--include-uncommitted`
   apply): a SHA-256 + byte length over the exact bytes `GeneratePatch` returns
   for that dir. Reuses the "generate once, reuse those bytes" flow already in
   `ApplyAll` (`copyflow/apply.go:96-103`: `GeneratePatch` → `CheckPatch` →
   `ApplyPatch`, same `patchBytes` throughout) — the fingerprint check slots in
   between generation and `CheckPatch`, comparing the freshly generated bytes
   against the stored value, then handing those same bytes to `CheckPatch`/
   `ApplyPatch`. No second generation, so no extra race window beyond the
   generate→check→apply sequence that already exists.

Plain `yoloai diff <spec>` (the default single-dir invocation, no `--ref`,
`--log`, or `--all`) computes and stores **both** fingerprints for that dir —
whichever `apply` mode the user later picks, there's a matching anchor to
check. `--ref`/`--log`/`--all` are exploratory views of a different or wider
scope than what `apply` would land right now, so **they never write the
fingerprint** — only the plain default `diff` does (confirmed: only the
diff that matches what `apply` would independently regenerate should be able
to clear the guard).

## Confirmed decisions

From the design conversation:

- **No prior `diff` run → no guard, falls through to the existing `[y/N]`
  prompt** (`internal/cli/cliutil/confirm.go`, gated by `--yes` as today).
  Skipping `diff` isn't evidence the user cares about drift — it's the
  ordinary "didn't bother to look" case, already handled.
- **Fingerprint present and mismatched → hard balk, unconditional, ignores
  `--yes`.** A drift means something changed that the user has not seen; `--yes`
  means "skip the routine confirmation," not "skip past an unreviewed change."
- **No `--force` bypass.** The way to clear a balk is to re-run `diff` — it
  regenerates and re-stores the fingerprint against current state. Reusing the
  verb that already shows the user what's new is simpler than adding a new flag,
  and it guarantees the user actually looked before the guard clears.
- **Fingerprint present and matches → no behavior change.** Same `[y/N]`
  prompt (or auto-proceed under `--yes`) as today.

## Data model

Add two optional fields to `store.DirEnvironment` (`store/environment.go:84`),
alongside `BaselineSHA`/`InceptionSHA`:

```go
// ReviewedCommitsHash is a hash of the ordered beyond-baseline commit SHA
// list as of the last plain `diff`, if any. Empty if never diffed (or only
// diffed via --ref/--log/--all).
ReviewedCommitsHash string `json:"reviewed_commits_hash,omitempty"`
// ReviewedPatchSHA256 / ReviewedPatchLen fingerprint the exact bytes
// GeneratePatch produced as of the last plain `diff`.
ReviewedPatchSHA256 string `json:"reviewed_patch_sha256,omitempty"`
ReviewedPatchLen    int64  `json:"reviewed_patch_len,omitempty"`
```

All `omitempty`, additive — likely doesn't need a schema version bump the way
the v1→v2 `Dirs` collapse did (`IncludeIgnored`/`StripHistory` were added the
same way), but confirm against the current `migrate()` convention before
building.

On a **successful** apply that advances `BaselineSHA`
(`AdvanceBaselineCAS`, `copyflow/apply.go:604`), clear all three fields for
that dir — the reviewed diff has now been landed, so the fingerprint no
longer describes "unreviewed content" and should not linger.

## Where the check lands

- **Default apply (`ApplyModeCommits` → `copyflow.ApplySeries`,
  `apply.go:146`):** after `ListCommitsBeyondBaseline` produces the current
  commit list, hash it and compare to `ReviewedCommitsHash`. Mismatch → balk.
  No stored value → skip the check.
- **`--no-commit` (`copyflow.ApplyAll`, `apply.go:68`):** after
  `GeneratePatch` returns `patchBytes` (`apply.go:96`, before `CheckPatch`),
  compare `len(patchBytes)`/`sha256(patchBytes)` against
  `ReviewedPatchLen`/`ReviewedPatchSHA256`. Mismatch → balk.
- **`--patches` (export):** doesn't land anything, no check needed.
- **`apply --all`:** loops per tracked dir (`internal/cli/workflow/apply.go:346`
  path) — each dir's own fingerprint(s) apply independently, same as
  everything else in the multi-dir model (per
  [multi-workdir-diff-apply.md](multi-workdir-diff-apply.md)).

## Open questions

- **Does `diff`'s default scope already match `apply`'s default scope?**
  `diffSingle` runs `git add -A` then `git diff <BaselineSHA>` — a worktree-
  vs-baseline comparison that includes uncommitted changes. The default
  (commit-preserving) `apply` mode lists *commits* beyond baseline and applies
  a format-patch series — it's not obviously clear it picks up uncommitted
  changes at all. If the two already have different scopes today, that's a
  pre-existing UX question independent of this guard, but it determines
  whether the commit-scope fingerprint alone is sufficient for the common case
  or whether plain `diff` needs to warn/split when uncommitted changes exist
  outside what commit-scope covers. Needs verification against
  `copyflow/apply.go`'s commit-preserving path before implementation.
- **Should `diff --all` also write per-dir fingerprints?** It already loops
  per-dir `GenerateDiff` calls under the hood, so this would be cheap to add.
  Deferred here to keep the "only the plain single-dir diff sets the
  fingerprint" rule simple; revisit if `--all` review is a common pre-apply
  step in practice.
- **Messaging on balk:** exact wording, but should clearly say *why* ("the
  sandbox has changed since you last ran `diff`") and *what to do*
  ("run `yoloai diff <spec>` to review, then `apply` again").

## Out of scope (v1)

- Any bypass flag for a mismatched guard. Re-running `diff` is the only path.
- Extending the guard to `reset` or other workdir-mutating verbs — this is
  specifically about the diff-review → apply gap.
